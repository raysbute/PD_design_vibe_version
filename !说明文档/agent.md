## 使用C/C++实现一个适用于小型医院的轻量级的医疗管理系统。

### 基本要求（这一部分是必做的）
1. 医疗管理系统至少容纳100名普通门诊患者，30名住院患者，20名医生，5个科室（每个科室至少3名医生），3种病房（部分科室与病房相关），至少20类药品（部分药品与科室相关）；且以上情况需要考虑人名重名、存储长度，药品名长度、商品名、通用名、别名等因素的影响。
2. 医疗管理系统至少包含：
   患者信息和医疗记录管理：患者个人信息、相关联的医疗记录，医疗记录至少包括“挂号、看诊、检查、住院”4类和此类医疗记录的必要信息；
   住院病房和床位管理：至少包含病房类型和关联科室、床位状态管理，患者与床位的关联；
   药房和药品管理：药品出入库管理、处方发药管理。
3. 医疗管理系统要能够提供各种报表，考虑不同情况下、不同权限人员需要查看的报表信息：如按照患者、医生、科室、药品名等基础信息查询；适用于医护视角、管理视角和患者视角的统计查询。
4. 程序必须有良好的健壮性，可以应对输入数据类型错误，部分数据缺失的情况，**绝对不允许出现程序意外退出的情况。**
5. 所有核心数据结构均需要用链表实现，实体之间的关系通过ID匹配来实现，来避免C++复杂的指针，ID统一使用STL string实现。
6. 在程序运行结束时，保存当前所有数据，在程序关闭后再次启动时，可以恢复程序上次关闭前的状态。如果不存在文件或者文件格式有误，应该从默认初始状态启动。
7. 程序支持使用txt文件或者标准输入输出流输入调试。

### 项目约定
1. 所有人员，药品，科室名称不超过20个字符，规定每个病房最多有30个床位。
2. 使用一个全局变量unsigned long long globalTime（单位：秒）来标记全局时间，通过改变这个变量来在答辩时模拟时间流逝，初始为0，使用一个全局变量unsigned int day表示现在是第几天，时间格式为 Dayx HH-MM-SS，使用一个全局变量week表示现在星期几，规定第一天是星期一，用来明确医生什么时候上班
3. 使用一个全局变量 double money 表示医院现有的资金，精确到小数点后2位，初始可以设置为0或10000.00。money 只受以下三项影响：患者支付的挂号费（固定为20元）、药品费（按处方药品总价计算）、住院费（200元/天，按住院天数从押金中扣缴），不考虑采购成本和检查费。
4. 部分关系示意：
    1. 科室-医生 一对多
    2. 科室-病房 一对多，科室可以不对应病房，也可以对应多个
    3. 科室-药品 多对多，一个科室对应多种药品，一种药品也可以由多个科室来使用
    4. 病房-住院患者 一对多，病房下的病床可以用指针数组实现，每一个病床可以指向一个患者
    5. 医生-住院患者 多对多
    6. 医生-门诊患者 一对多
    7. 医生-处方 一对多
    8. 患者-处方 一对多
    9. 患者-诊疗记录 一对多
5. 尽量保证程序代码易于阅读。

### 详细实体说明
1. 科室 (Department)
    属性：科室ID（唯一）、科室名称、科室描述
    关联：医生列表（通过医生中的科室ID关联）、病房列表、药品列表
2. 医生 (Doctor)
    属性：医生ID/工号（唯一）、姓名、级别（主任医师/副主任医师/主治医师/住院医师）、所属科室ID、出诊星期（一个或多个星期几，如1,3,5表示周一三五出诊）
    关联：负责的门诊患者列表（通过挂号记录关联）、负责的住院患者列表、处方列表
3. 患者 (Patient)
    属性：患者ID（唯一，可用身份证号）、姓名、年龄、联系方式、患者类型（门诊/住院）
    关联：挂号记录列表、处方列表、诊疗记录列表
    1. 门诊患者 (Outpatient) — 继承或包含Patient基础信息
        额外关联：当前挂号信息
    2. 住院患者 (Inpatient) — 继承或包含Patient基础信息
        额外属性：住院押金余额（double，精确到小数点后2位）、入院天数、所在床位ID
        额外关联：责任医生列表
4. 挂号记录 (Registration)
    1. 属性：挂号ID（唯一，编码需隐含日期和当天顺序号，如 R-YYMMDD-序号）、患者ID、科室ID、医生ID、挂号天数（记录时的 timeDay）、状态（待就诊/已就诊/已过期/已撤销）
    2. 该记录在患者挂号时创建，标志患者进入医生的工作队列
5. 看诊记录 (Consultation)
    属性：看诊ID（唯一）、挂号ID（关联）、患者ID、医生ID、主诉、诊断、看诊天数、状态（有效/已撤销）
    该记录由医生接诊后创建，一次挂号对应至多一条看诊记录
6. 检查记录 (Examination)
    属性：检查ID（唯一）、看诊ID（关联，检查依附于某次看诊）、患者ID、检查项目名称、费用、检查天数、状态（有效/已撤销）
    可以由门诊看诊或住院期间开具
7. 处方记录 (Prescription)
    属性：处方ID（唯一）、看诊ID（关联，处方依附于某次看诊）、患者ID、医生ID、药品列表（药品ID+数量+单价）、总金额、处方天数、状态（有效/已撤销）
    开处方时不扣库存，仅在患者缴费后扣减对应药品库存
8. 住院记录 (Hospitalization)
    属性：住院ID（唯一）、患者ID、病房ID、床位号、责任医生ID列表、入院天数、出院天数（0表示未出院）、住院押金余额、状态（在院/已出院/已撤销）
    住院费用每日扣除（按 timeDay 推进时由系统自动从押金中扣200元），押金不足时可继续住院（扣为负数表示欠费），出院时结算
9. 药品 (Medicine)
    属性：药品ID（唯一）、商品名、通用名、别名、规格、库存量（int）、消耗量累计（int）
    关联：可使用的科室列表

10. 病房 (Ward)
    属性：病房ID（唯一）、病房类型（特殊病房/普通病房）、所属科室ID、床位数量（≤30）、床位状态列表（每个床位：占用/空闲）
    关联：住院患者列表（通过床位关联）

### 功能
系统根据不同权限的人员可以给出不同的功能。

1. 管理员
    1. 登录：通过管理员账号和密码登录系统。
    2. 科室管理：增加/修改/删除科室，给科室分配病房，给药品分配科室。
    3. 病房管理：增加/修改/删除病房，更改病房归属科室。
    4. 医生管理：增加/修改/删除医生，修改医生归属科室、出诊星期。
    5. 药品管理：增加/修改/删除药品，修改药品信息。
    6. 患者管理：增加/修改/删除门诊和住院患者的基本信息。
    7. 挂号管理：为患者办理挂号（指定科室、医生，检查医生当日是否出诊及挂号是否已满），生成挂号记录。
    8. 住院管理：为患者办理住院登记（分配病房和床位），生成住院记录并收取初始押金；办理出院结算（释放床位，结算剩余押金）。
    9. 药房管理：药品入库（增加库存）、药品出库（减少库存），查看药品库存和消耗量。
    10. 报表查询（详见下方【报表功能】）
2. 医生
    1. 通过医生ID登录系统
    2. 查看待诊患者：列出挂号为本人且状态为“待就诊”的所有门诊患者，按挂号顺序排列。
    3. 接诊：选择一个待诊患者，创建看诊记录（填写主诉、诊断），将挂号状态改为“已就诊”。
    4. 开具处方：为正在看诊的患者开处方，选择药品及数量（检查库存），生成处方记录和药品费用，处方关联到当前看诊记录。
    5. 开具检查：为正在看诊的患者开检查单，生成检查记录和费用，检查关联到当前看诊记录。
    6. 查看我的住院患者：列出本人作为责任医生的所有在院患者，可查看其诊疗记录、处方等。
    7. 为住院患者开处方/检查：功能同上，记录关联到患者住院期间的治疗。
3. 患者
    1. 可以输入患者ID来登录,系统自动识别门诊/住院患者身份。
    2. 查看记录：查看本人的所有诊疗记录（挂号、看诊、检查记录、处方、住院记录）。
    3. 支付费用：查看待缴费项目（挂号费、处方药费），逐项或一次性支付。支付后医院 money 增加，对应处方药品库存减少，住院患者直接从押金扣除（押金不足则提示补缴）。
    4. 住院功能（仅住院患者）：查看住院信息（病房、床位、责任医生）、当前押金余额和每日费用明细。

### 报表查询
1. 管理员
    某科室/全院下所有药品，可按剩余量、消耗量、名称字典序排序，可按药品名/ID搜索
    某科室/全院下所有医生信息，可按ID排序
    某科室下属病房当前床位占用情况
    全部/某患者的完整就诊记录
    统计医院当前资金（money）及构成
    统计每位医生在某段时间内的接诊量和处方数（工作繁忙程度）
    统计某段时间内的营业总额
2. 医生
    本人某段时间内的门诊患者列表及对应诊疗记录
    本人负责的住院患者当前情况
    本人开具的处方统计
3. 患者
    本人的所有就诊记录、处方、费用明细
    如果是住院患者，查看当前住院信息和费用

### 健壮性要求
程序必须能应对但不限于以下异常情况，不允许崩溃：
    输入数据类型错误：如需要整数却输入字符串、需要字符却输入过长字符
    输入数据超出范围：如日期数值超出约定天数、金额为负数、年龄为负数/超大值
    ID 引用错误：引用的患者ID、医生ID、科室ID等在对应链表中不存在
    逻辑冲突：如为同一患者在同一天同一科室重复挂号、分配已占用的床位、给不可用科室的医生开药
    部分数据缺失：录入记录时字段不完整，需给出明确提示并要求重新输入
    链表操作边界：链表为空时执行删除/查询、遍历到尾节点后的操作
    文件读写异常：文件不存在、文件格式错误、文件内容不完整，需给出错误提示并使用默认初始状态

### 链表操作要求
程序应充分体现链表基本操作，包括但不限于：
    增加：头插法/尾插法/按ID字典序插入
    删除：按ID查找后从链表中摘除节点（物理删除）或标记为作废状态（逻辑删除，用于诊疗记录的撤销操作）
    修改：查找节点后修改数据域内容，对于诊疗记录应采用“撤销+新增”方式
    查询：遍历链表、按条件筛选、多链表关联查询
