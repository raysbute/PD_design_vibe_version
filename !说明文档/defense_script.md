#### 一、全局基础设施设计

首先是 `global.h` 和 `global.cpp`，这两个文件定义了系统的四个核心全局变量：

- **`globalTime`** —— 全局秒数，模拟世界的时间流逝，0 表示 Day1 的 00:00:00
- **`day`** —— 绝对天数，从 globalTime 自动计算 `day = globalTime / 86400 + 1`
- **`week`** —— 星期几，从 day 自动计算 `week = ((day - 1) % 7) + 1`（1=周一..7=周日）
- **`money`** —— 医院资金，所有金额操作经过 `round2()` 保留两位小数

这里有一个关键设计：我写了一个 **`setTime()` 函数作为时间的唯一修改入口**：

```cpp
void setTime(unsigned long long newTime) {
    globalTime = newTime;
    day = (unsigned int)(newTime / 86400) + 1;
    week = ((day - 1) % 7) + 1;
}
```

这意味着 day 和 week 永远由 globalTime 推导得出，不存在两份数据不一致的问题。整数除法的向下取整特性保证了——不满 86400 秒（一天），day 不会变化，也就不会触发住院扣费。

同时，我定义了所有常量：最大病床数 30、挂号费 20 元、住院费每天 200 元、一天 86400 秒。还有 **10 种 ID 前缀**（DEPT、DOC、PAT、R-、CS、EX、PR、MED、WRD、HSP-、ADM），以及**多套状态码体系**——挂号的"待就诊/已就诊/已过期"、处方的"待缴费/已撤销/已缴费"、住院的"在院/已出院"等。这些常量让整个项目有统一的"语言"。

#### 二、实体结构体设计

接下来是 `entities.h`，我设计了 **12 个实体结构体**，每个都包含 `next` 指针，构成单向链表节点：

| 序号 | 结构体 | 用途 | 关键设计 |
|------|--------|------|----------|
| 1 | DepartmentNode | 科室 | deptID + name + description |
| 2 | DoctorNode | 医生 | level(0-3) + departmentID + workDays[7] |
| 3 | PatientNode | 患者 | type区分门诊/住院 + deposit + bedID |
| 4 | RegistrationNode | 挂号 | regID含日期 + patientID/doctorID/deptID 三向关联 |
| 5 | ConsultationNode | 看诊 | regID一对一关联 + complaint/diagnosis |
| 6 | ExaminationNode | 检查 | consultID关联 + itemName + cost |
| 7 | PrescriptionNode | 处方 | totalAmount + status(待缴费/已缴费/已撤销) |
| 8 | MedicineNode | 药品 | stock库存 + consumed累计消耗 |
| 9 | PrescMedicineNode | 处方-药品关联 | prescID + medID + quantity + unitPrice |
| 10 | DeptMedicineNode | 科室-药品关联 | deptID + medID |
| 11 | WardNode | 病房 | bedStatus[30] 定长数组管理床位 |
| 12 | HospitalizationNode | 住院记录 | doctorIDs[10] + admitDay/dischargeDay + deposit |
| 13 | AdminNode | 管理员 | adminID + password |

我有几个特别想强调的设计：

**第一，统一链表管理门诊和住院患者**。PatientNode 通过 `type` 字段（0=门诊/1=住院）区分，共用一条链表。查找患者时不需要去不同地方找，一条链表遍历即可。

**第二，用 ID 字符串做关联，而不是指针**。医生的 `departmentID` 字段存的是 "DEPT001" 这个字符串，而不是指向 DepartmentNode 的指针。这样的好处是——删除一个科室时，医生的数据不会出现悬空指针。我们只需要在删除前遍历检查有没有关联的医生就行。

**第三，多对多关系通过中间表实现**。PrescMedicineNode 实现了处方和药品的多对多关系——一个处方可以有多种药品，同一种药品也可以出现在多个处方中。DeptMedicineNode 同理实现了科室和药品的多对多。

#### 三、链表操作层

`list_ops.h` 和 `list_ops.cpp` 是系统的**数据访问层**，约 960 行代码。我为每一种实体都实现了标准化的链表操作函数：

**三种插入方式**：头插法 O(1)、尾插法 O(n)、字典序插入 O(n)。不同的实体用不同的插入策略——挂号记录用尾插保证先后顺序（先挂号的先看诊），医生用字典序插入方便查找。

**物理删除操作**：分为删除头节点和删除中间/尾节点两种情况处理：
```cpp
// 删除头节点
if (head->key == target) {
    NodeType* tmp = head;
    head = head->next;
    delete tmp;
}
// 删除中间节点——找到前驱，跳过目标节点
cur->next = targetNode->next;
delete targetNode;
```

**ID 生成系统**是我另一个重要设计。10 个全局计数器（`nextDeptID`、`nextDoctorID` 等），加上格式化的生成函数：
```cpp
string generateDeptID() {
    char buf[32];
    snprintf(buf, sizeof(buf), "DEPT%03d", nextDeptID++);
    return string(buf);
}
```
挂号ID和住院ID还额外嵌入了日期信息，如 `R-0001-016` 表示 Day1 的第 16 个挂号。还有一个 `updateIDCountersFromLists()` 函数，在从文件加载数据后自动同步计数器，确保新生成的 ID 不会与已有 ID 冲突。

此外，我还为 **WardNode 实现了床位管理**的三个专用函数：`getFreeBedCount`（统计空床位）、`getAllocateBed`（分配第一个空闲床位，写入患者ID）、`releaseBed`（清空指定床位）。

#### 四、鲁棒输入/输出

`utils.h` 和 `utils.cpp` 是我写的**输入防线**。我一共实现了 7 个输入函数，每个都遵循同一个模式：

```cpp
int readInt(const char* prompt) {
    int val;
    while (true) {
        cout << prompt;
        if (cin >> val) {
            clearInputBuffer();
            return val;
        }
        cin.clear();           // 清除 failbit
        clearInputBuffer();    // 丢弃错误数据
        cout << "[错误] 输入不是有效整数，请重新输入。" << endl;
    }
}
```

核心原理是 **cin 的流状态管理**。当用户输入 "abc" 时，`cin >> val` 会设置 failbit，之后任何 cin 操作都会失败。必须先用 `cin.clear()` 清除错误标志，再用 `cin.ignore()` 吃掉缓冲区里的错误数据。

在这个基础上，我做了分层封装：
- `readInt` → `readIntRange(prompt, min, max)` 增加了范围校验
- `readDouble` → `readNonNegDouble(prompt)` 增加了非负校验
- `readString(prompt, maxLen)` 使用 `getline` 读取整行，去除首尾空白，校验非空和长度

还有一个 `readConfirm` 函数，用于所有删除、修改、出院等关键操作的确认，接受 y/yes/是 或 n/no/否。

#### 五、主程序框架

`main.cpp` 只有 150 行，但它是整个系统的**骨架**。我在这里声明了全部 13 个链表头指针（全部初始化为 nullptr），实现了启动流程和主菜单循环。

启动时先尝试加载存档，失败则回退到默认数据初始化：
```cpp
if (!loadAllData(...)) {
    setTime(0);
    money = 10000.00;
    initDefaultData(...);
}
```

主菜单循环是一个 `while(true)` 结构，根据用户选择分发到管理员、医生、患者或批量导入模块。退出时调用 `cleanupAll()` 遍历 13 条链表，逐一释放所有动态分配的内存，确保没有内存泄漏。

#### 四、管理员功能

`menus_admin.cpp` 约 1860 行，是我工作量最大的模块，实现了 **11 项管理功能**。

**登录系统**：3 次尝试机会，ID 和密码双重验证。

**科室管理**（6个子功能）：增删改查 + 分配病房 + 分配药品。删除科室时有一个关键的**外键检查**——遍历 DoctorNode 统计关联医生数，遍历 WardNode 统计关联病房数，有任何关联则拒绝删除。这是保证数据完整性的关键防线。

**医生管理**（4个子功能）：添加医生时，出诊日的解析最复杂——用户输入逗号分隔的 "1,3,5"，我用 `stringstream + getline(ss, token, ',')` 解析，对每个值去首尾空白后 `atoi` 转为整数，校验范围 1-7，去重后存入 `workDays[]`。

**挂号管理**——这是最复杂的业务逻辑，涉及多重校验：
1. 验证患者存在
2. 验证科室存在
3. 筛选当日出诊的该科室医生（`week → doctorWorksOnDay`）
4. **防重复挂号**：同患者+同科室+同日+待就诊状态，四条件匹配

**住院管理**——入院流程包括：
1. 验证患者为门诊类型
2. 筛选同科室病房
3. 显示床位占用状况（遍历 `bedStatus[]` 数组）
4. 选择空闲床位
5. 多选主管医生（最多10名）
6. 输入押金 → 修改患者类型 → 创建住院记录

出院结算设计上，因为每日费用已在时间推进中扣除，出院时直接使用当前押金余额作为退款金额。

**时间推进**——这是系统的核心自动化机制：

```cpp
for (unsigned int d = 0; d < daysAdvanced; d++) {
    setTime(globalTime + SECONDS_PER_DAY);
    // 逐日扣住院费：deposit -= 200, money += 200
    for (h = hospHead; h; h = h->next)
        if (h->status == 0) { h->deposit -= 200; money += 200; }
    // 同步患者记录、检查过期挂号
}
```

逐日循环是关键——住院费按完整天数扣除，不足一天不计费。挂号过期的判断是 `regDay < day`，确保当天的挂号不会被误标为过期。

**支付费用**——核心财务功能：
1. 收集待缴费项：遍历 RegistrationNode（status == PENDING → 20 元/笔）、PrescriptionNode（status == 0 → 处方金额）
2. 三个支付选项：全部支付/仅支付挂号费/指定处方支付
3. **门诊患者**：医院直接收款 `money += amount`
4. **住院患者**：从押金扣除 `hosp->deposit -= amount; money += amount`
5. 支付后自动扣减药品库存：遍历 PrescMedicineNode → `med->stock -= pm->quantity; med->consumed += pm->quantity`

#### 三、报表查询系统

`reports.cpp` 约 1070 行，实现了**三个入口**下的共 **10 种报表**。

**管理员报表**（7种）：
- **药品查询**：支持按科室筛选，5 种排序方式（库存/消耗量升序降序、名称字典序），大小写不敏感搜索。使用 `std::sort` + lambda 表达式实现排序
- **医生信息查询**：按科室筛选，显示级别和出诊日
- **病房床位占用**：遍历 `bedStatus[]`，空闲显示"空闲"，已占用调用 `findPatientByID` 解析患者名
- **患者就诊记录**：遍历 5 条链表聚合全部诊疗信息
- **资金统计**：分挂号费、药费、住院费三项统计
- **医生工作量**：遍历 ConsultationNode 和 PrescriptionNode 统计每位医生的接诊量和处方数，按总量降序排名
- **营业总额**：指定时间范围统计，住院天数与时间范围的交集计算使用重叠区间算法 `overlapStart = max(admitDay, startDay); overlapEnd = min(dischargeDay, endDay)`

**医生报表**（3种）：门诊患者统计、住院患者统计、处方统计（按已支付/未支付/已取消分组）。

**患者报表**（1种）：个人就诊摘要 + 完整记录 + 当前住院信息。

所有报表的时间范围筛选算法是通用的：
```cpp
if (day >= startDay && day <= endDay) {
    // 记录在此时间范围内，纳入统计
}
```

## 整体总结

我们三人合作开发了这套**小型医院医疗管理系统**，项目总代码量约 7600 行（不含注释和空行则更少），实现了 12 个实体类型、3 种角色、21 项业务功能、10 种统计报表。

核心亮点可以归纳为：

| 序号 | 亮点 | 说明 |
|------|------|------|
| 1 | **纯链表实现** | 12 类实体全部使用手动管理的单向链表，无 STL 容器 |
| 2 | **ID 关联解耦** | 实体间通过 string ID 关联，避免指针复杂性 |
| 3 | **统一时钟** | globalTime 唯一入口 setTime()，day 和 week 自动推导 |
| 4 | **三角色分离** | 管理员 11 项、医生 7 项、患者 3 项，功能视图不同 |
| 5 | **完整医疗流程** | 挂号 → 接诊 → 处方/检查 → 缴费 → 住院 → 出院 |
| 6 | **自动化机制** | 时间推进自动扣费、过期挂号、批量导入 |
| 7 | **金融准确性** | 所有金额经 round2() 处理，防浮点误差 |
| 8 | **崩不溃** | 7 个输入函数保证任何输入不崩溃 |
| 9 | **数据持久化** | 14 个文件保存，转义处理，异常回退到默认数据 |
| 10 | **多维度报表** | 管理员 7 种 + 医生 3 种 + 患者个人统计 |
| 11 | **内存安全** | 所有 new 有对应 delete，退出时完整清理 |
| 12 | **批量导入** | 8 种实体、管道分隔格式、文件/stdin 双模式 |

以上是我们的答辩，感谢各位老师。
